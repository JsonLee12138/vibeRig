# 开发前生命周期与职责

## 阶段门禁

| 阶段 | 负责人 | 输入 | 必须产出 | 退出条件 |
|---|---|---|---|---|
| 需求基线 | PM / 产品经理 | 老板原始需求、项目现状 | `intake.md`、`requirement.yaml` | 老板确认目标、范围、约束和成功信号 |
| PRD 决策 | 产品负责人 | 需求基线、已有 PRD | 关联、新建或不需要 PRD的结论 | 决策及理由已记录 |
| 领域调研 | 各专业负责人 | 统一上下文包 | 领域报告、可行性结论、必要 spike | 关键未知已验证或变成显式风险 |
| 架构综合 | CTO / 架构师 | PRD、调研报告、代码现状 | 架构、契约、ADR、红白队记录 | 关键争议已裁决，方案可实现和可测试 |
| 质量设计 | QA + 各专业负责人 | 需求与架构 | 验收、测试、追踪、老板验证指南 | 所有目标可观测、可验证；TC 有权威执行阶段与环境 |
| 交付规划 | 研发负责人 / PM | 架构、验收、风险、测试 | Milestone / Issue 本地草案 | 依赖、顺序、责任、验证方式明确 |
| DoR 审核 | CTO | 全部开发前产物 | `pre-development-review.md` | DoR 全部通过或例外获授权 |
| 老板审批 | 老板 | CTO 汇总包 | 批准、条件批准或退回 | 批准范围和条件留痕 |

## 领域研究维度

| 角色 | 必查维度 | 典型产出 |
|---|---|---|
| 产品 | 用户旅程、业务规则、权限角色、异常流程、非目标、成功指标 | 场景矩阵、规则冲突、指标定义 |
| 前端 | 页面/状态边界、数据获取、缓存、性能、可访问性、兼容性、可视化状态 | 组件边界、状态模型、交互失败模式 |
| 后端 | 领域模型、API、事务、一致性、并发、幂等、容量、错误语义 | 服务边界、接口与数据方案 |
| 数据 | schema、迁移、回填、质量、保留、血缘、分析口径 | 数据契约、迁移与校验方案 |
| 安全/合规 | 鉴权、授权、敏感数据、审计、威胁、法规 | 威胁与控制、风险接受项 |
| 运维/SRE | 部署、配置、可观测性、容量、降级、回滚、灾备 | 发布与运行手册、SLO/告警 |
| QA | 测试层级、边界、负向、兼容、性能、回归、E2E | 测试策略、用例与证据要求 |

只路由受影响角色。低风险需求可以合并角色，但不得省略相关维度。

## 红白队与 CTO 裁决

1. 白队基于调研提出推荐方案和备选方案。
2. 红队独立攻击边界、契约、数据、安全、失败模式、迁移、运维、成本和交付计划。
3. 白队逐条回应：接受、部分接受或反驳，并给证据。
4. CTO 将每条发现标为 `resolved`、`accepted_risk`、`owner_decision` 或 `rework`。
5. `owner_decision` 仅用于权限性问题；技术偏好不得向上甩锅。

## Definition of Ready

进入开发前必须全部满足：

- 需求目标、用户、范围、非目标、约束、成功指标已确认；
- PRD 决策及理由已记录；
- 关键技术、外部依赖和现状均完成可行性验证；
- 推荐架构、接口、数据、权限、失败模式、迁移和回滚已明确；
- 红白队关键问题已关闭或获风险授权；
- 每个结果有验收标准、老板验证步骤和工程验证方式；
- 测试覆盖单元、集成、契约、E2E、回归及适用的非功能测试；每条 TC 明确自动化、权威执行阶段、环境、风险和证据要求；
- 发布、配置、可观测性、告警、回滚与运维责任已定义；
- 风险有负责人、触发信号、缓解和兜底；
- Milestone / Issue 草案覆盖范围、依赖、验收项与测试；
- 追踪表不存在孤立的需求、验收项、关键测试或交付 Issue；跨 Issue/UAT TC 明确归属 Milestone；
- CTO 汇总包包含推荐结论和老板需要决策的最小事项。

任何不适用项必须写明理由，不能静默跳过。

## 返工策略

| 变更 | 最小重跑范围 |
|---|---|
| 文案或非功能性说明 | 验收、测试、汇总包 |
| 业务规则或范围 | PRD 决策之后全部相关阶段 |
| 接口/数据/权限设计 | 架构、验收、测试、风险、交付规划 |
| 外部依赖结论 | 对应领域调研及所有下游引用 |
| 老板附加审批条件 | 条件涉及的阶段 + DoR + 汇总包 |

保留旧结论、变更原因和受影响对象，不整套无差别重跑。
