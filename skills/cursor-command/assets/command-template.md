# Cursor Command Templates

使用说明：复制对应模板到 `.cursor/commands/<name>.md`，填写 `[...]` 占位符后删除本文件头部说明。

---

## 模板 A：代码审查（review-code）

```markdown
---
name: review-code
description: 全面审查当前文件的质量、bug 和安全问题
---

审查当前打开的文件，逐项检查以下维度：

**Bug**
- 逻辑错误、边界条件、空指针/未定义访问
- 异步竞态、错误处理缺失

**性能**
- 算法复杂度（标出 O(n²) 以上的循环）
- 不必要的重复计算、内存泄漏风险

**安全**
- 输入验证缺失、SQL/命令注入风险
- 硬编码凭证、不安全的随机数

**可维护性**
- 函数过长（>50 行）、命名不清晰、重复代码

每个问题输出：
- 行号
- 问题描述（一句话）
- 建议修复（代码片段或方向）
```

## 模板 B：PR 描述生成（create-pr-description）

```markdown
---
name: create-pr-description
description: 基于 git diff 生成结构化 PR 描述
---

基于当前分支与 main 的差异（@git diff main），生成 PR 描述：

## 变更内容（What）
- 用 bullet 列出主要改动，每条 ≤ 20 字

## 变更原因（Why）
- 说明业务背景或技术动机

## 测试方案（Testing）
- [ ] 单元测试覆盖新逻辑
- [ ] 手动测试步骤（如有 UI 变更）
- [ ] 回归风险点

## 注意事项（Notes）
- breaking change / 配置变更 / 迁移步骤（如无则删除此节）
```

## 模板 C：Bug 修复（fix-bug）

```markdown
---
name: fix-bug
description: 分析当前文件中的 bug 并给出最小化修复方案
---

分析当前文件中报告的 bug（用户会在此命令后补充描述）：

1. **定位根因**：找到引发问题的具体行，解释为什么会出错。
2. **最小修复**：给出改动最少、副作用最小的修复方案，不要顺带重构。
3. **验证方法**：说明如何验证修复有效（测试命令或手动步骤）。
4. **潜在风险**：列出此修复可能影响的其他模块（如有）。
```

## 模板 D：生成单元测试（generate-tests）

```markdown
---
name: generate-tests
description: 为当前文件生成单元测试
---

为当前打开的文件生成单元测试：

- 遵循项目已有的测试框架和风格（先读 @codebase 中的测试文件样例）
- 覆盖：正常路径、边界值、错误场景
- 每个测试用 Arrange / Act / Assert 注释标注结构
- 不要为 private 实现细节写测试，只测公开行为
- 如有 mock 需求，与现有 mock 模式保持一致

输出：完整可运行的测试文件，路径与当前文件同级，文件名加 `.test` 或 `_test` 后缀。
```

## 模板 E：重构建议（suggest-refactor）

```markdown
---
name: suggest-refactor
description: 识别当前文件的重构机会并给出优先级排序
---

分析当前文件，识别重构机会，按收益/风险比排序：

**高优先级（建议立即处理）**
- 重复代码（DRY 违反，提取函数/常量）
- 过长函数（>50 行，拆分职责）
- 魔法数字/字符串（提取为具名常量）

**中优先级（下次迭代处理）**
- 命名不清晰的变量/函数
- 嵌套过深的条件（提前返回/卫语句）

**低优先级（可选）**
- 注释更新
- 格式统一

每条建议附上：当前代码位置（行号）+ 改后示意。
不要输出完整重构后的文件，只输出建议清单。
```

## 模板 F：部署检查（pre-deploy-check）

```markdown
---
name: pre-deploy-check
description: 部署前执行安全和配置检查清单
---

在部署当前变更前，检查以下项：

**环境配置**
- [ ] `.env` 文件未被提交（检查 `.gitignore`）
- [ ] 所有环境变量已在目标环境配置
- [ ] 数据库迁移脚本已准备并测试

**代码质量**
- [ ] CI 流水线全绿
- [ ] 没有 `console.log` / `debugger` 残留
- [ ] 没有硬编码的测试数据或 API key

**依赖**
- [ ] 无已知 CVE 漏洞（运行 `npm audit` 或等效命令）
- [ ] lock 文件已更新并提交

**回滚计划**
- 描述如果部署失败的回滚步骤

输出：逐项检查结果（✅/❌），❌ 项给出修复建议。
```
