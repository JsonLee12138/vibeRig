# 规则内容编写指南

## 应该写什么

- 项目特定约定（linter 覆盖不到的）
- 命名、结构、模式偏好
- 架构边界（哪些模块禁止互相依赖）
- 业务逻辑约束
- 工作流程指导（提交规范、PR 格式）
- 常见错误的预防性说明

## 不应该写什么

| 反模式 | 原因 |
|--------|------|
| 完整 ESLint / Prettier 配置 | 委托给 linter |
| 工具的完整文档 | AI 已知常见工具用法 |
| 极少见的边缘情况 | 降低信噪比 |
| 代码库已有的注释内容 | 冗余 |
| 宽泛约束如"写好代码" | 无指导意义 |

## 规则正文结构

推荐结构（按需取用）：

```
## 核心约定
[3-10 条具体指令，每条一行]

## 示例

### 正确
[代码片段]

### 错误
[代码片段，说明原因]

## 例外情况
[确实存在例外时才写此节]
```

## 质量检查

- 每条指令是否能直接指导 AI 行为？
- 有没有"写好"这类空洞词汇？
- 是否提供了正/反示例？
- 全文是否 < 500 行？

具体规则模板见 `assets/rule-templates.md`。
