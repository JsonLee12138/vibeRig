# VibeRig 开发前流程总体设计

## 文档信息

| 属性 | 值 |
|---|---|
| 状态 | Proposed |
| 版本 | 1.0 |
| 日期 | 2026-07-21 |
| 适用范围 | VibeRig 新需求的需求发现与开发前准备流程 |
| 不适用范围 | Bug 快速修复、小型维护任务、开发执行、PR 合并、开发后验收 |

## 1. 背景

VibeRig 当前已经具备需求记录、技术调研、架构设计、验收标准、里程碑拆分、Issue 拆分、开发执行和验收等独立 Skills，但现有流程要求用户理解并逐个触发多个阶段。

这与真实企业研发协作存在偏差。在企业中，老板或业务负责人通常只负责提出目标、补充业务事实、批准研发方案和最终验收；产品、研发、架构、测试、安全和运维团队应在内部完成调研、讨论、评审和交付规划，然后由 CTO 统一向老板汇报。

本设计将新需求流程重构为两个用户参与阶段：

1. 产品经理通过 Intake 与老板完成需求发现和需求基线确认。
2. AI 研发部门完成全部开发前工作后，由 CTO 提交开发前方案包，请老板统一批准。

两个阶段之间的 PRD 判断、技术调研、可行性分析、架构设计、红白队评审、验收设计、测试设计、发布设计、风险分析和交付拆分全部由 AI 内部自动完成。

## 2. 设计目标

| 目标 | 说明 |
|---|---|
| 单一人工入口 | 新需求只需要人工执行 Intake，不要求用户了解后续 Skill 名称或顺序 |
| 完整需求基线 | Intake 以产品经理方式完成业务访谈，而不是只记录一句需求 |
| 自动开发前编排 | Intake 确认后自动启动完整开发前流程，不要求用户逐阶段推动 |
| 多专业并行分析 | 根据需求影响面动态启动前端、后端、数据、安全、QA、SRE 和领域专家 |
| CTO 统一裁决 | 各 Subagent 提供专业证据，最终方案由主 Agent 以 CTO 身份综合和裁决 |
| 对抗性评审 | 初版方案必须经过红队攻击、白队答辩和 CTO 裁决 |
| 可执行验收 | 开发前即定义老板如何配置、运行、观察并判断结果 |
| 全链路追踪 | 用户故事、AC、测试用例、交付任务和风险之间建立稳定映射 |
| 明确开发边界 | 开发前流程只交付批准过的开发资料，不侵入 task-runner 等开发阶段职责 |

## 3. 非目标

本设计不改变以下开发阶段能力的职责：

- `task-runner`：执行已批准的 Linear Issue 或 Milestone。
- `agent-sop`：编排开发、验证、QA 和返工。
- `accept-issue`：执行 Issue 级验收。
- `accept-milestone`：执行里程碑级验收、合并和归档。
- `merge-issue`：合并独立 Issue 的 PR。
- `subagent-routing`：提供通用的能力路由和有边界的 Subagent Brief。

本设计也不要求所有小改动和 Bug 进入完整开发前流程。它们继续使用已有快速通道。

## 4. 核心决策

| 编号 | 决策 |
|---|---|
| D-01 | Intake 是新需求唯一需要人工主动参与的入口 |
| D-02 | Intake 结束时先由老板确认需求基线，再自动进入开发前流程 |
| D-03 | 是否创建 PRD 由 AI 根据需求规模和产品结构自动判断 |
| D-04 | `tech-research` 从用户工作流入口降级为研发部门内部调研协议 |
| D-05 | 开发前流程由新的 `pre-development` 编排能力负责 |
| D-06 | CTO 是开发前流程的唯一整合者和最终技术裁决者 |
| D-07 | 专业 Subagent 不直接形成最终架构，不更新 Linear，不向老板单独汇报 |
| D-08 | 开发前方案必须经过红队攻击、白队答辩和 CTO 裁决 |
| D-09 | 验收标准必须同时包含业务结果、老板验收流程和工程验证方法 |
| D-10 | 老板批准前只生成本地 Milestone/Issue 草案，不创建正式 Linear 结构 |
| D-11 | 老板批准开发前方案后，才固化文档并将交付计划写入 Linear |
| D-12 | 开发阶段直接消费批准后的开发前资料包，不重复进行产品和架构决策 |

## 5. 参与角色与职责

| 角色 | 承担者 | 主要职责 | 明确禁止 |
|---|---|---|---|
| 老板 / Product Owner | 用户 | 提出目标、提供业务事实、确认需求基线、接受重大取舍、批准开发 | 不负责组织技术调研，不逐个触发内部 Skills |
| 产品经理 / BA | Intake 主 Agent 或产品 Subagent | 需求访谈、用户故事、业务规则、范围、非目标、成功指标 | 不替老板编造业务决策 |
| CTO / Delivery Owner | 主 Agent | 组队、编排、综合、跨领域裁决、开发就绪判断、老板汇报 | 不把 Subagent 原始报告直接当作最终方案 |
| 前端负责人 | 专业 Subagent | 前端现状、交互架构、状态、性能、兼容性和可访问性 | 不决定后端或数据领域最终方案 |
| 后端负责人 | 专业 Subagent | 服务边界、API、并发、一致性、缓存、幂等和失败恢复 | 不绕过 CTO 单独确定跨域契约 |
| 数据负责人 | 专业 Subagent | 数据模型、索引、迁移、回填、兼容、保留和回滚 | 不忽略历史数据和回滚路径 |
| 安全负责人 | 专业 Subagent | 威胁模型、鉴权、越权、输入、敏感数据和供应链风险 | 不接受未记录的安全风险 |
| QA 负责人 | 专业 Subagent | 可测试性、测试策略、测试用例、环境、Fixture 和覆盖矩阵 | 不以“功能正常”代替可执行验证 |
| SRE / DevOps | 专业 Subagent | 部署、容量、可观测性、Feature Flag、灰度和回滚 | 不把合并 PR 等同于上线成功 |
| 领域专家 | 动态 Subagent | 支付、计费、合规、搜索、AI 等领域规则 | 不超出明确领域范围 |
| 红队 | 独立评审 Subagents | 只寻找方案缺陷、失败条件和隐藏假设 | 不负责替原设计者修复方案 |
| 白队 | 原领域负责人 | 回应红队发现、提供证据、修订或合理反驳 | 不允许无证据地拒绝发现 |

## 6. 人机协作边界

### 6.1 正常人工接触点

| 接触点 | 用户动作 | AI 动作 |
|---|---|---|
| Intake 访谈 | 回答产品和业务问题，纠正错误理解 | 读取已有上下文，逐步访谈并收敛需求 |
| 需求基线确认 | 确认或修改需求摘要 | 固化 `intake.md` 和 `requirement.yaml` |
| CTO 开发前汇报 | 批准、有条件批准、要求修改或延期 | 根据决策固化方案、定向返工或停止 |

### 6.2 允许中途打断用户的情况

开发前内部流程默认不中断用户。只有以下情况可以升级给老板：

- 存在两个互斥的业务方向，AI 无权决定产品取舍；
- 涉及预算、采购、外部供应商或商业合同；
- 涉及不可逆的数据删除、迁移或重大兼容性破坏；
- 需要老板明确接受安全、隐私、合规或业务风险；
- 时间、范围和质量之间存在无法内部化解的冲突；
- 缺少的信息会让后续全部分析失去意义，而不是只影响某个局部方案。

其他不确定项由 CTO 采用可逆的保守假设继续分析，并记录在假设清单和最终汇报中。

## 7. 总体流程

| 阶段 | 主责 | 输入 | 输出 | 用户参与 |
|---|---|---|---|---|
| 1. Intake | 产品经理 | 老板的原始需求、项目上下文 | 需求基线 | 是 |
| 2. PRD 决策 | CTO + 产品经理 | 需求基线、现有 PRD | PRD 关联、PRD 草案或无需 PRD 的理由 | 否 |
| 3. 风险与组队 | CTO | 需求基线、仓库现状 | 风险等级、领域矩阵、研究计划 | 否 |
| 4. 多领域调研 | 专业 Subagents | 领域 Brief、相关代码和文档 | 标准化领域报告 | 否 |
| 5. CTO 综合 | CTO | 全部领域报告 | 跨域约束、推荐方案、初版架构 | 否 |
| 6. 红白队评审 | 红队、白队、CTO | 初版架构 | 发现、答辩、修订和裁决记录 | 否 |
| 7. 验收与测试设计 | 产品经理、QA、技术负责人 | 需求基线、最终架构 | AC、老板验收流程、Test Plan、Test Cases | 否 |
| 8. 发布与风险设计 | SRE、数据、安全 | 最终架构、NFR、测试计划 | 发布、迁移、回滚、可观测性和风险登记 | 否 |
| 9. 交付拆分 | CTO、QA、技术负责人 | 全部批准候选文档 | Milestone/Issue 草案、追踪矩阵 | 否 |
| 10. DoR 审核 | CTO | 完整开发前资料包 | Ready、Conditional 或 Not Ready | 否 |
| 11. CTO 汇报 | CTO | 审核后的资料包 | 老板可决策的开发前报告 | 是 |
| 12. 方案固化 | CTO | 老板决策 | 正式文档、Linear 计划、开发交接 | 仅反馈决策 |

## 8. Intake 设计

### 8.1 定位

Intake 不是“记录一句需求”，而是产品经理代表研发组织与老板完成需求发现。它必须把模糊目标收敛成后续团队可以独立工作的业务契约。

Intake 应先读取项目现有文档、历史需求、已有关联 PRD 和代码上下文，避免向老板询问 AI 可以自行发现的问题。

### 8.2 必须覆盖的访谈维度

| 维度 | 必须获得的信息 |
|---|---|
| 背景 | 为什么现在提出，当前成本或机会是什么 |
| 目标用户 | 谁使用、谁管理、谁审批、谁受影响 |
| 核心问题 | 当前具体哪里不可用或效率不足 |
| 期望结果 | 用户获得什么新能力，业务得到什么结果 |
| 主流程 | 用户从哪里进入，按什么顺序完成任务 |
| 异常流程 | 输入错误、权限不足、依赖失败和中断时如何处理 |
| 业务规则 | 状态、限制、计算、审批和决策规则 |
| 权限模型 | 谁能看、谁能创建、谁能修改、谁能审批 |
| 范围 | 本期必须交付的能力 |
| 非目标 | 本期明确不做的内容 |
| 优先级 | P0、P1、P2 或等价优先级 |
| 成功指标 | 可观察、可量化的业务结果 |
| 时间成本 | 截止时间、预算和外部承诺 |
| 兼容迁移 | 老用户、历史数据、旧接口和旧行为要求 |
| 风险偏好 | 快速交付、稳定性、安全性和完整性之间的偏好 |
| 验收主体 | 谁最终验收，使用什么环境、账号和设备 |
| 验收关注点 | 老板最关注的数据、流程、页面和可视化结果 |
| 参考材料 | 竞品、截图、文档、现有流程和示例数据 |

### 8.3 访谈原则

- 一次聚焦一个主题，问题附带基于现有证据的最佳猜测；
- 不重复询问仓库、文档或配置已经能够回答的问题；
- 不把技术实现选择伪装成业务问题交给老板；
- 对“现代化、易用、快速、稳定”等模糊描述继续追问可观察标准；
- 对老板未明确但可逆的细节记录假设，不无限追问；
- 访谈记录不直接落盘，只写收敛后的需求结论。

### 8.4 需求基线确认

Intake 完成后向老板展示一份需求理解摘要，至少包含：

- 问题与目标；
- 用户和主要场景；
- 业务主流程与异常流程；
- 业务规则和权限；
- 本期范围与非目标；
- 成功指标；
- 兼容、时间和风险约束；
- 老板验收主体与关注点；
- 已采用的假设和仍然开放的问题。

老板确认后，需求状态进入 `requirement_baselined`，并自动启动开发前流程。

## 9. PRD 自动判定

### 9.1 判定规则

| 条件 | 处理 |
|---|---|
| 单一、边界清晰的用户能力 | 不创建 PRD，直接使用 Requirement |
| 一个目标包含多个独立能力 | 创建 PRD，并拆出多个关联 Requirements |
| 涉及多个用户群或用户旅程 | 通常创建 PRD |
| 需要在多个能力之间做 P0/P1/P2 排序 | 创建 PRD |
| 跨多个发布周期或版本 | 创建 PRD |
| 属于已有产品计划 | 关联已有 PRD |
| Bug、技术债、小型维护 | 不创建 PRD |

### 9.2 行为约束

- 主流程中不询问老板“是否运行 prd-brainstorm”；
- PRD 所需事实优先来自已确认的 Intake；
- 如果 Intake 信息足够，自动生成 PRD 草案，不重复进行相同访谈；
- 如果 PRD 引入了 Intake 中没有的业务选择，将其记录为老板决策项；
- PRD 随 CTO 开发前方案包一起批准，而不是单独设置人工门禁；
- 用户显式要求产品级脑暴时，`prd-brainstorm` 仍可保留独立人工访谈模式。

## 10. 风险识别与动态组队

CTO 根据需求影响面选择必要角色，不机械启动全部 Subagents。

| 信号 | 必须参与的角色 |
|---|---|
| 页面、组件、交互、跨端 | 前端、UI/UX、QA |
| API、服务、任务、第三方集成 | 后端、QA、安全 |
| Schema、查询、事件、历史数据 | 数据、后端、QA |
| 鉴权、权限、敏感数据 | 安全、后端、QA |
| 部署、容量、环境、可用性 | SRE、后端、QA |
| 公共接口或跨团队契约 | 架构、后端、消费者代表、QA |
| 支付、计费、合规等专业领域 | 对应领域专家、安全、QA |
| 多模块或不可逆变更 | 全面架构评审和红白队 |

轻量需求允许一个 Subagent 承担多个相邻领域，但输出仍必须按领域结构分开，避免遗漏。

## 11. 内部技术调研协议

`tech-research` 在本设计中不是人工阶段，而是每个专业 Subagent 在形成建议前必须遵守的调查协议。

### 11.1 通用研究步骤

1. 检查当前实现、项目约定和历史决策。
2. 明确事实、假设和未知项。
3. 对版本敏感内容查询官方资料或一手来源。
4. 识别至少一个可行方案和一个有意义的替代方案。
5. 必要时执行只读实验、原型或 Spike。
6. 评估可行性、复杂度、风险、依赖和测试影响。
7. 以标准化领域报告返回 CTO。

### 11.2 各领域研究重点

| 领域 | 研究重点 |
|---|---|
| 前端 | 组件与路由、状态归属、请求模式、设计系统、浏览器兼容、渲染性能、可访问性、错误状态、测试能力 |
| 后端 | 服务边界、API 契约、并发、幂等、事务、一致性、缓存、重试、超时、限流、降级、第三方依赖 |
| 数据 | Schema、索引、读写路径、历史数据、迁移、回填、兼容、保留、备份和回滚 |
| 安全 | 资产、信任边界、认证、授权、越权、输入验证、注入、密钥、隐私、审计和依赖风险 |
| QA | AC 可测试性、测试分层、环境、Fixture、Mock 边界、E2E、负向路径和回归范围 |
| SRE | 部署拓扑、容量、SLO、日志、指标、追踪、告警、Feature Flag、灰度和回滚 |
| 性能 | 延迟预算、吞吐、并发、内存、网络、数据库瓶颈、基准数据和压力模型 |
| 领域 | 领域规则、外部标准、法律限制和专有失败语义 |

### 11.3 标准化领域报告

每个专业 Subagent 返回以下字段：

| 字段 | 内容 |
|---|---|
| Current State | 当前系统如何工作 |
| Evidence | 代码路径、配置、官方资料和实验结果 |
| Constraints | 已知约束和不可改变项 |
| Options | 可行方案及其差异 |
| Recommendation | 推荐方案和理由 |
| Feasibility | 可行、有条件可行或不可行 |
| Interfaces | 与其他领域的输入输出契约 |
| Risks | 风险、触发条件、影响和缓解方式 |
| Test Impact | 需要如何验证 |
| Operational Impact | 部署、监控和运维影响 |
| Open Questions | 当前无法内部解决的问题 |
| Complexity | 工作量来源和复杂度等级 |

Subagent 不直接更新正式需求文档、Linear 状态或最终架构。它们只提交有证据的部门报告。

## 12. CTO 综合与架构设计

CTO 不能简单拼接部门报告，必须解决跨领域冲突并形成一个统一系统方案。

### 12.1 CTO 综合职责

- 统一产品目标和技术约束；
- 识别前端、后端、数据、安全、QA 和 SRE 之间的依赖；
- 为每个状态确定唯一属主；
- 为每个接口确定调用方、提供方、数据格式和错误语义；
- 解决不同部门建议之间的冲突；
- 明确构建、采购、复用或放弃的决策；
- 形成推荐方案、替代方案和权衡理由；
- 将业务风险与纯技术风险分开；
- 只把需要老板权限的决策升级给用户。

### 12.2 架构设计必须覆盖

| 设计面 | 必须内容 |
|---|---|
| 系统边界 | 本期改变什么，不改变什么 |
| 模块职责 | 每个模块的职责和非职责 |
| 接口契约 | 参数、返回、错误、版本和兼容策略 |
| 数据模型 | 实体、关系、状态、索引和生命周期 |
| 数据流 | 数据来源、处理、持久化和消费路径 |
| 权限模型 | 主体、资源、动作和授权边界 |
| 失败模式 | 超时、重试、部分失败、并发冲突和恢复 |
| 迁移方案 | Schema、历史数据、回填、双写或切换策略 |
| 可观测性 | 日志、指标、追踪、告警和业务监控 |
| 发布策略 | Feature Flag、灰度、兼容窗口和回滚 |
| 可测试性 | 测试接口、Fixture、Mock 边界和验证点 |
| 决策记录 | 采用、拒绝和延期方案的理由 |

## 13. 红白队对抗评审

### 13.1 评审角色

| 评审组 | 攻击目标 |
|---|---|
| 架构红队 | 边界模糊、耦合、扩展性、接口版本和职责遗漏 |
| 失败模式红队 | 超时、重试、并发、部分失败和数据不一致 |
| 安全红队 | 越权、注入、隐私、密钥、滥用和供应链 |
| 交付红队 | 不可测试、不可迁移、不可观测、不可发布和不可回滚 |

### 13.2 评审过程

1. CTO 发布不含内部辩护过程的初版设计。
2. 红队独立攻击方案，只报告问题，不修复方案。
3. 原领域负责人组成白队，对分配给自己的问题提供证据和响应。
4. 白队对有效问题修订方案，对无效问题给出可验证的拒绝理由。
5. CTO 对全部发现进行最终分类。
6. 阻塞级问题修复后可以进行一次定向复审。
7. 同类阻塞问题两轮仍无法解决时，升级为老板决策项或判定 Not Ready。

### 13.3 发现分类

| 分类 | 含义 |
|---|---|
| `accepted-fixed` | 问题成立，方案已修订 |
| `accepted-mitigated` | 问题成立，通过限制、监控、灰度或流程缓解 |
| `accepted-risk` | 问题成立，需要老板明确接受残余风险 |
| `rejected-with-evidence` | 问题不成立，附有证据 |
| `deferred` | 不属于本期，已记录后续处理边界 |

## 14. 验收设计

### 14.1 三层验收结构

| 层级 | 回答的问题 | 面向对象 |
|---|---|---|
| 业务验收标准 | 什么结果表示需求完成 | 老板、产品经理 |
| 老板验收流程 | 老板如何配置、运行、查看并判断 | 老板、业务验收人员 |
| 工程验证 | 研发如何通过测试、命令、日志和数据证明正确 | CTO、研发、QA |

### 14.2 每条 AC 的结构

| 字段 | 内容 |
|---|---|
| `id` | 稳定编号，例如 `AC-1` |
| `title` | 一句话业务能力 |
| `source` | Intake 或 PRD 中的来源 |
| `precondition` | 验收开始前的系统状态 |
| `action` | 验收者执行的业务动作 |
| `expected` | 可判定的业务结果 |
| `owner_verification` | 老板可照做的验收流程 |
| `engineering_verification` | 自动化测试、命令、日志和数据检查 |
| `expected_data` | 正常数据的精确标准 |
| `expected_visual` | 正常页面和可视化标准 |
| `failure_signals` | 明确的不通过条件 |
| `evidence` | 截图、录像、输出、日志或 API 响应 |
| `cleanup` | 验收后清理数据和恢复配置的方法 |
| `milestone` | AC 归属里程碑，拆分前为空 |

### 14.3 老板验收流程结构

每条需要业务验收的 AC 必须包含：

| 部分 | 要求 |
|---|---|
| 验收目标 | 该流程证明哪项业务能力 |
| 环境 | 本地、测试、Staging 或受控生产环境 |
| 地址 | 页面、系统或接口入口 |
| 账号权限 | 测试角色和账号获取方式 |
| 前置配置 | Feature Flag、环境变量和第三方配置；不得记录真实密钥 |
| 测试数据 | 数据准备、Seed 或选择条件 |
| 启动步骤 | 需要启动的服务和计划提供的命令 |
| 操作步骤 | 按顺序可直接执行的动作 |
| 数据标准 | 字段、数量、状态、精度、排序和一致性 |
| 可视化标准 | 页面、组件、图表、状态、文案和交互表现 |
| 失败标准 | 出现哪些情况必须判定不通过 |
| 证据留存 | 截图、录像、导出文件或其他证据 |
| 清理方法 | 验收后恢复环境的方法 |

### 14.4 数据标准

“数据正常”不是合法验收标准。数据标准必须说明：

- 数据来源；
- 预期数量；
- 关键字段的精确值或允许范围；
- 金额、比例、时间和小数精度；
- 正常、处理中、完成和失败状态；
- 排序、过滤和分页规则；
- 重复提交后的去重或幂等结果；
- 页面、API、数据库和导出之间的一致性；
- 操作后允许的数据可见延迟；
- 空值、零值、缺失值和边界值表现。

### 14.5 可视化标准

可视化验收必须说明：

- 页面入口和测试视口；
- 正常、加载、空数据、错误和无权限状态；
- 必须出现的表格、卡片、按钮和图表；
- 标题、提示和错误文案；
- 数据与图表轴、图例、单位和数据点的映射；
- 成功、警告和失败的状态表达；
- 筛选、排序、翻页、悬停和点击行为；
- 响应式布局和允许的滚动行为；
- 键盘、焦点、对比度和读屏要求；
- 设计稿、截图或组件规范等视觉基准。

### 14.6 验收流程生命周期

| 状态 | 含义 |
|---|---|
| `planned` | 开发前已定义环境、步骤、预期和计划提供的命令 |
| `implemented` | 开发已提供验收页面、脚本、Seed 和测试能力 |
| `verified` | QA 已在目标环境实际执行并记录结果 |
| `accepted` | 老板或授权验收人完成验收并接受结果 |
| `rejected` | 验收失败，记录失败步骤和实际结果 |

开发前报告中的命令必须标记为计划提供，不能把尚未实现的命令描述为已经可执行。

## 15. 测试设计

QA 在两个时机参与：

| 时机 | 责任 |
|---|---|
| 需求基线阶段 | 检查业务规则是否可验证，补充异常路径、边界条件和行为 AC |
| 架构定稿阶段 | 将 AC 转换为单元、契约、集成、E2E、性能、安全和恢复测试 |

### 15.1 测试类型

| 类型 | 主要目标 |
|---|---|
| 单元测试 | 函数、状态转换、计算和边界逻辑 |
| 契约测试 | API、事件和模块接口 |
| 集成测试 | 数据库、队列、缓存和第三方服务协作 |
| E2E 测试 | 关键用户路径和跨模块行为 |
| 探索测试 | 发现需求与计划之外的问题 |
| 性能测试 | 延迟、吞吐、并发和资源预算 |
| 安全测试 | 权限、注入、泄露和滥用路径 |
| 恢复测试 | 超时、重试、部分失败、迁移和回滚 |
| 可访问性测试 | 键盘、语义、焦点、对比度和读屏 |
| 生产 Smoke | 发布后关键路径和基础健康状态 |

### 15.2 测试计划必须包含

- 测试范围和明确不测的范围；
- AC 到 Test Case 的完整映射；
- 每种测试层级的职责，避免重复覆盖；
- 环境、账号、权限、Fixture 和 Seed；
- Mock 与真实依赖的边界；
- 自动化与人工验证的划分；
- 通过标准、失败标准和阻塞等级；
- 回归范围和性能基线；
- 验收证据的保存方式。

## 16. 发布、迁移与可观测性设计

| 领域 | 必须回答的问题 |
|---|---|
| 发布 | 如何部署，是否使用 Feature Flag 或灰度 |
| 兼容 | 新旧客户端、接口和数据如何共存 |
| 迁移 | Schema、历史数据和配置如何变更 |
| 回填 | 数据如何补齐，如何校验完整性 |
| 回滚 | 代码、数据和配置分别如何恢复 |
| 容量 | 峰值、并发和资源预算是否满足 |
| 日志 | 关键行为和失败是否可定位 |
| 指标 | 技术指标和业务指标如何观察 |
| 告警 | 什么条件触发告警，由谁响应 |
| Smoke | 发布后立即执行哪些关键验证 |
| 观察期 | 多久后才能判定发布稳定 |

## 17. 交付拆分与追踪

开发前流程在老板批准前只生成本地交付草案。

### 17.1 拆分原则

- Milestone 按用户价值和可发布增量划分；
- 架构模块和依赖约束实施顺序，但不机械等同于 Milestone；
- Issue 是可独立实现、测试和提交的垂直切片；
- 每个 Issue 必须映射至少一个 AC 或风险控制项；
- QA、集成和发布工作优先嵌入功能切片，确需独立治理时才单独建项；
- 未经老板批准，不创建正式 Linear Milestone 或 Issue。

### 17.2 全链路追踪

| 来源 | 必须追踪到 |
|---|---|
| 业务目标 | 成功指标 |
| 用户故事 | 行为 AC |
| 行为 AC | 老板验收流程和 Test Cases |
| 非功能需求 | 性能、安全、恢复和可观测性测试 |
| 架构决策 | 受影响的 Milestone 和 Issues |
| 风险 | 缓解措施、验证方式和责任 Issue |
| Issue | Commit、PR 和 Proof Packet |
| PR | Review、CI 和发布版本 |
| 发布版本 | 生产 Smoke、监控和最终验收证据 |

## 19. 开发就绪门禁

### 19.1 DoR 检查

| 门禁 | 通过条件 |
|---|---|
| 需求门禁 | 需求基线已由老板确认，范围、非目标和成功指标清晰 |
| PRD 门禁 | 已关联、已生成，或已有无需 PRD 的明确理由 |
| 研究门禁 | 所有受影响领域有证据充分的可行性结论 |
| 架构门禁 | 模块、接口、数据、权限、失败、迁移和可观测性完整 |
| 对抗门禁 | 红白队完成；无未处理阻塞发现 |
| 验收门禁 | 每个目标有 AC；老板验收流程可照做；数据和视觉标准明确 |
| 测试门禁 | AC-Test 映射完整；环境、Fixture 和通过标准明确 |
| 风险门禁 | 高风险项有缓解、接受或升级路径 |
| 发布门禁 | 部署、灰度、迁移、回滚、监控和 Smoke 已规划 |
| 交付门禁 | Milestone/Issue 草案覆盖需求、测试和风险 |
| 追踪门禁 | 目标、AC、测试、任务和风险不存在孤立项 |

### 19.2 审核结论

| 结论 | 含义 |
|---|---|
| Ready | 开发前资料完整，可以向老板申请批准开发 |
| Conditional | 可以汇报，但必须明确批准条件和满足方式 |
| Not Ready | 存在阻塞缺口，不应请求老板批准开发 |

## 20. CTO 开发前汇报

`pre-development-review.md` 是面向老板的主文档，详细领域报告作为附件。

### 20.1 汇报结构

1. 需求理解与业务目标；
2. PRD 判定和产品范围；
3. 本期范围与非目标；
4. 推荐方案的高层说明；
5. 主要备选方案及放弃理由；
6. 用户流程和关键业务规则；
7. 架构、接口、数据和权限的核心决策；
8. 可行性结论和主要证据；
9. 红白队发现及处理摘要；
10. 验收标准摘要；
11. 老板验收手册；
12. 测试和质量策略；
13. 发布、迁移、回滚和监控方案；
14. 高风险项和缓解措施；
15. Milestone/Issue 草案和复杂度来源；
16. 需要老板决定或接受的事项；
17. CTO 的 Ready、Conditional 或 Not Ready 结论。

### 20.2 老板验收手册摘要

汇报必须明确告诉老板：

- 验收使用哪个环境和地址；
- 需要什么账号、角色和权限；
- 需要先配置哪些变量、开关或第三方能力；
- 需要运行哪些计划提供的服务、Seed 或验证命令；
- 从哪个页面开始，按什么顺序操作；
- 每一步应看到什么数据、状态和页面表现；
- 正常数据的数量、数值、精度、排序和一致性标准；
- 图表、表格、卡片、状态和异常页面应如何呈现；
- 出现哪些情况必须判定不通过；
- 需要保存哪些截图、录像、导出或日志证据；
- 验收后如何清理数据和恢复配置。

### 20.3 老板决策

| 决策 | 后续动作 |
|---|---|
| 批准开发 | 固化文档，创建 Linear Milestone/Issue，交给开发流程 |
| 有条件批准 | 记录条件；条件满足并验证后自动进入开发 |
| 要求修改 | 只重跑受影响领域、CTO 综合和对应评审 |
| 延期或拒绝 | 保存资料，不创建正式开发任务 |

## 21. 状态模型

| 状态 | 含义 |
|---|---|
| `intake` | 正在进行需求访谈 |
| `requirement_baselined` | 老板已确认需求理解 |
| `pre_development` | AI 研发部门正在执行开发前流程 |
| `pre_development_blocked` | 存在必须由老板或外部条件解决的阻塞 |
| `awaiting_owner_approval` | CTO 已提交开发前方案包 |
| `conditionally_approved` | 老板有条件批准，条件尚未全部满足 |
| `ready_for_development` | 方案已批准且 Linear 计划已固化 |
| `planned` | 已交接现有开发流程使用 |

状态变更必须有对应文档或决策证据，不能只更新枚举值。

## 22. 返工与恢复

老板要求修改时，不从 Intake 或全部调研重新开始。CTO 应执行影响分析后定向重跑：

| 修改类型 | 重跑范围 |
|---|---|
| 文案或展示调整 | 产品、前端、QA、验收手册 |
| 业务规则调整 | 产品、受影响技术领域、QA、交付计划 |
| 数据模型调整 | 数据、后端、安全、QA、迁移和发布计划 |
| 权限调整 | 产品、后端、安全、QA |
| 技术方案调整 | 受影响领域、CTO 综合、红白队、测试计划 |
| 范围扩大 | PRD 判定、风险组队、受影响的全部后续产物 |
| 发布约束调整 | SRE、数据、安全、QA 和交付计划 |

所有返工必须保留变更原因、受影响文档和重新验证结果。

## 23. 与现有 Skills 的关系

| Skill | 设计后的定位 | 计划变化 |
|---|---|---|
| `intake` | 唯一人工需求入口 | 扩展访谈维度；需求确认后自动启动开发前流程 |
| `prd-brainstorm` | 显式产品脑暴入口 + 内部 PRD 综合能力 | 主流程内部调用时不重复访谈 |
| `tech-research` | 内部领域调研协议 | 移除“用户主动触发”的主流程要求 |
| `architecture-design` | CTO 综合、架构定稿和红白队裁决 | 从用户脑暴改为消费领域报告 |
| `define-acceptance` | 内部验收设计能力 | 取消逐 AC 人工门禁；增加老板验收与工程验证结构 |
| `split-milestones` | 开发前交付草案与批准后固化能力 | 老板批准前不写 Linear |
| `split-issues` | Issue 草案与追踪映射能力 | 增加 Test Case、风险和架构契约映射 |
| `subagent-routing` | 通用底层路由 | 职责不变，由开发前编排复用 |
| `task-runner` | 开发执行 | 开发前职责边界不变；测试消费、证据和门禁由[开发质量门禁设计](./development-quality-gates.zh-CN.md)细化 |
| `agent-sop` | 开发、QA 和返工协议 | 开发前职责边界不变；风险路由与去重由开发质量门禁设计细化 |
| `accept-issue` | 可选的单 Issue 验收 | 开发前职责边界不变；普通里程碑 Issue 不再强制逐个验收 |
| `accept-milestone` | 里程碑验收、合并和归档 | 开发前职责边界不变；采用证据复用、增量回归、E2E 与老板 UAT |
| `pre-development` | Intake 后的开发前总编排 | 新增 |

## 24. 方案比较

| 方案 | 结论 | 原因 |
|---|---|---|
| 用户逐个调用 Intake、Research、Architecture、Acceptance、Split | 拒绝 | 用户必须理解研发内部流程，人工门禁过多，阶段容易遗漏 |
| 把所有逻辑继续塞入 Intake | 拒绝 | Intake 同时承担产品、技术、测试和交付职责，难以维护且职责失焦 |
| Intake + 自动 Pre-development + CTO 汇报 | 采用 | 人工边界清晰，内部专业分工完整，用户只做业务和批准决策 |
| 所有专业 Subagent 分别向老板汇报 | 拒绝 | 信息碎片化、冲突未解决，老板被迫承担 CTO 职责 |
| 老板批准后再做架构和测试设计 | 拒绝 | 老板无法基于完整成本、风险和验收方式做开发决策 |

## 25. 后续实施建议

实施应按以下顺序进行，避免同时重写全部 Skills：

1. 定义 `pre-development` 的输入、输出、状态和 DoR 契约；
2. 扩展 Intake 访谈和需求基线结构；
3. 定义领域研究报告模板与 CTO 综合模板；
4. 将 `tech-research` 调整为内部调研协议；
5. 调整 `architecture-design` 为 CTO 综合与红白队流程；
6. 升级验收 Schema 和老板验收手册模板；
7. 增加 Test Plan、Test Case、Risk Register、Release Plan 和 Traceability Schema；
8. 为 Milestone/Issue 拆分增加草案模式和批准后固化模式；
9. 建立场景评测，验证流程选择、Subagent 覆盖、人工中断和产物完整性；
10. 在不改变职责的前提下验证现有开发 Skills 能消费新的资料包。

## 26. 设计验收标准

本设计实施后应满足：

- 用户提出新需求后，只需完成 Intake 访谈和需求基线确认；
- Intake 确认后，开发前流程自动运行，不要求用户逐个调用内部 Skills；
- AI 能自动判断 PRD 是关联、创建还是不需要，并给出理由；
- 受影响的专业领域均有 Subagent 调研报告，无关领域不会机械启动；
- CTO 能解决跨领域冲突并产出统一架构，而不是拼接报告；
- 架构经过红队、白队和 CTO 裁决，所有发现都有最终分类；
- 每个业务目标都有 AC，每个 AC 都有老板验收流程和工程验证方法；
- 老板验收流程包含配置、运行、查看、数据标准、可视化标准、失败标准和证据；
- 每个 AC 映射到 Test Case 和交付草案，不存在孤立目标、测试或任务；
- 老板批准前不创建正式 Linear Milestone/Issue；
- CTO 汇报能让老板基于范围、方案、风险、成本、测试、验收和发布信息做一次决策；
- 方案批准后可以无产品歧义地交给现有开发流程；
- `task-runner`、`agent-sop`、`accept-issue` 和 `accept-milestone` 不承担开发前需求与架构决策；其开发质量门禁按[开发质量门禁与验收流程设计](./development-quality-gates.zh-CN.md)执行。

## 27. 影响与权衡

### 正向影响

- 用户不再承担研发流程编排职责；
- 技术方案在开发前得到多领域证据支持；
- 产品、架构、测试、发布和验收形成统一资料包；
- 重大风险在投入开发前暴露；
- 验收方式提前确定，减少开发完成后的预期偏差；
- 开发阶段可以直接消费稳定契约，减少重复分析。

### 成本与风险

- 开发前流程会增加 AI 计算和 Subagent 调用成本；
- 如果领域路由过度，会造成不必要的报告和分析；
- 如果 Intake 需求基线不完整，后续自动流程可能放大错误假设；
- 如果 CTO 只拼接报告而不解决冲突，资料完整但方案仍不一致；
- 如果验收流程在实现前写得过于具体，可能描述不存在的命令，因此必须区分 `planned` 与 `verified`；
- 如果开发前资料缺乏结构化 Schema，后续追踪容易再次依赖自然语言推断。

上述风险通过动态组队、统一报告契约、红白队评审、DoR 门禁、结构化追踪和老板统一批准进行控制。
