# agent-sop

## 目标

创建一个 Codex skill，用于让主 agent 在当前项目中按能力路由型 SOP 编排 subagent 和 MCP 工具，完成任务分析、测试决策、测试编写与验证、开发、运行验证、QA 验收和失败回环。

主 agent 是流程控制器，负责拆解任务、选择合适 subagent、定义交付物、运行关键验证、整合结果、判断失败回环和最终汇报。

## 非目标

- 不新增 MCP server 功能。
- 不强制固定 subagent 名称。
- 不要求所有任务都必须写单测。
- 不替代主 agent 的工程判断。
- 不把该 skill 做成自动执行器；第一版是可被 Codex 遵循的 SOP 提示词协议。

## 候选需求

1. 当用户要求实现、修复、重构、验证或交付任务，且任务需要多阶段执行或受益于 subagent/MCP 编排时，主 agent 应使用 `agent-sop`。
2. 主 agent 必须先分析任务范围、风险、文件边界、验收标准和是否需要单测。
3. subagent 由主 agent 根据当前项目可用能力动态选择，例如 frontend、backend、ui、qa、security、db、docs、worker、reviewer 等。
4. skill 只规定阶段职责和验收闭环，不写死具体 subagent 名称。
5. 是否需要单测是强制判断点；是否实际写单测由任务风险和可测性决定。
6. 需要单测时，主 agent 先派合适 subagent 编写或调整测试，再进入开发。
7. 单测必须由 QA 能力或最接近验收能力的 subagent 验证覆盖度、边界和回归风险。
8. 开发阶段由主 agent 选择合适 subagent 执行，并明确任务边界、文件/模块 ownership、不得回退他人改动和预期验证方式。
9. 开发完成后，主 agent 使用当前 MCP/本地工具运行相关单测、构建、lint 或其他必要验证。
10. 最终必须由 QA 能力 subagent 或替代验收角色检查需求覆盖、测试证据、回归风险和残留问题。
11. 任一阶段验收失败，主 agent 必须带证据回到对应产出阶段循环修正。
12. 最终回复必须包含完成内容、验证结果、跳过项及理由、残留风险。

## 流程

`分析任务 -> 判断是否需要单测 -> 如需要则派合适 subagent 写单测 -> QA 验证单测 -> 派合适 subagent 开发 -> 主 agent 运行验证 -> QA 最终验收 -> 交付总结`

失败回环规则：

- 单测验收失败：回到测试产出 subagent。
- 开发验证失败：回到开发 subagent。
- 最终 QA 失败：由主 agent 判断回到分析、测试或开发阶段。
- 默认最多循环 3 轮。
- 如果同类偏差重复、方向明显错误或同一证据连续失败，可以在 2 轮后升级为阻塞。
- 如果每轮都有明确收敛，主 agent 可以说明理由后继续。
- 阻塞时必须汇总失败证据、已尝试方案和需要用户决策的点。

## 约束

- `agent-sop` 只定义主 agent 的执行协议，不绑定具体 MCP server 实现。
- QA 阶段不强制 subagent 名称必须为 `qa`。
- 主 agent 不能把整体任务完全外包后等待结果。
- 跳过单测必须记录理由。
- 所有失败回环都必须带证据，例如测试错误、验收意见、缺失覆盖点、文件路径或命令摘要。
- 大输出应使用 context-mode 类工具处理，只提取结论和关键错误。

## 假设

- Codex 当前运行环境支持 MCP 工具调用，并可能支持 subagent 调用。
- 不同项目会配置不同 subagent，因此不能假设固定角色集合。
- 主 agent 可以查看可用工具/subagent 描述，并据此做能力匹配。
- 第一版目标是稳定工作流提示词，不做自动状态机或代码层强制拦截。

## 依赖与来源

- 当前项目已作为 Codex 插件暴露 `skills/` 目录，适合新增 skill。
- 当前项目已有 MCP/context-mode 工具链，适合作为主 agent 验证和大输出处理通道。
- 需求来源为本次用户确认的能力路由型 SOP：主 agent 动态选择合适 subagent，SOP 只约束阶段、证据和失败回环。
